Synthèse

Le statu quo impose de recompiler N applications à chaque évolution d'EFF. Six familles techniques peuvent rompre ce couplage ; une seule s'aligne pleinement sur la stack Edifice.

Les apps Edifice sont construites avec Vite, agrégées par une couche Java/Vert.x, servies par Nginx depuis un dossier d'assets versionné par Jenkins. Toute solution retenue doit composer avec cet existant sans le réécrire.

Les blockers actuellement constatés sur la branche feat-ENABLING-722 — SVG en erreur runtime, duplications de React suspectées — sont des bugs identifiés, pas des limites architecturales.

→ Recommandation : Import Maps natives, standard W3C, déjà en place à 90 %.

Le contexte Edifice

Comprendre la douleur opérationnelle qu'on cherche à résoudre, et la stack technique avec laquelle il faut composer.

Le cycle de déploiement actuel

EFF est une librairie transverse publiée sur NPM, consommée par toutes les applications React de l'écosystème. À chaque évolution :

  1. Une nouvelle version de package est publiée.
  2. Toutes les applications consommatrices sont rebuildées via Jenkins pour intégrer la nouvelle version.
  3. Une remise en recette puis en production de l'ensemble est planifiée.

Ce couplage rend le moindre patch coûteux à propager. Une correction de 5 minutes dans EFF entraîne plusieurs heures de pipeline et de coordination de déploiement.

L'objectif

Permettre :

  • Le déploiement indépendant d'EFF (et d'explorer-front) sans rebuild des apps.
  • Le déploiement indépendant des applications (ex. : blog) sans dépendre du planning de la librairie.
  • La cohabitation ponctuelle de plusieurs versions d'EFF pendant des migrations.

Contraintes structurelles

  • Bundler : Vite 5 / Rollup, monorepo pnpm + Turbo côté EFF.
  • Runtime : React 18, navigateurs modernes uniquement (pas d'IE11).
  • Hébergement : Nginx servant /assets/js/* et /assets/importmap.json depuis un dossier versionné par Jenkins.
  • Agrégation : couche Vert.x assemble plusieurs SPA sur des chemins distincts (/blog, /explorer, etc.).
  • Assets : 483 SVG dans @edifice.io/bootstrap/dist/images, importés tantôt via svgr, tantôt comme URL.

Six approches passées en revue

Chaque famille est présentée par son principe technique, ses forces, et ses faiblesses spécifiques. Une seule reçoit la mention Recommandée.

Approche 01 / 06

Import Maps natives

★ Recommandée

Le standard W3C qui laisse le navigateur résoudre les import 'react' via un simple fichier JSON.

EFF est compilée en ESM puis servie depuis Nginx. Un fichier importmap.json central déclare la résolution : "react" → "/assets/js/edifice-react/react-esm/react.js". Au build, les apps marquent EFF et React comme externals ; à l'exécution, le navigateur charge directement les modules depuis l'URL indiquée par l'import map.

C'est l'approche déjà engagée sur la branche feat-ENABLING-722. La régression SVG observée est due à un changement de regex dans vite.config.ts ; elle se corrige en une ligne.

Forces

  • Standard W3C natif, supporté par tous les navigateurs modernes depuis 2023.
  • Zéro runtime ajouté : le navigateur fait le travail.
  • Indépendant du bundler : Vite aujourd'hui, n'importe quoi demain.
  • Une seule source de vérité pour les versions, partagée par toutes les apps.
  • Scopes natifs pour les versions multiples (/blog/ → version A, autres → version B).
  • Cache HTTP trivial, CDN-friendly.

Faiblesses

  • Tout doit être ESM : SVG, CSS, fonts à inliner ou pré-transformer au build d'EFF.
  • Mapping des sub-paths fastidieux à maintenir pour les nombreuses entrées d'exports.
  • Pas de code-splitting partagé entre apps.
  • Vitest ne consulte pas l'import map : alias dédié à prévoir pour les tests.
  • Aucune isolation : une régression dans EFF impacte instantanément toutes les apps.
Approche 02 / 06

Module Federation

Surdimensionnée

Le système de chunks fédérés de Webpack 5. Négociation au runtime entre hosts et remotes.

EFF expose explicitement ses modules ; chaque app les déclare comme remotes. Un share scope négocie au chargement quelle version de React utiliser si plusieurs versions cohabitent. Approche mature côté Webpack, beaucoup moins côté Vite : @originjs/vite-plugin-federation souffre d'un dev-mode instable et de types incomplets.

Forces

  • Share scopes intelligents pour le partage de dépendances singletons.
  • Granularité fine : on peut exposer un composant individuel.
  • Mature dans l'écosystème Webpack (Bytedance, Shopify, etc.).
  • Code-splitting dynamique hérité du bundler.

Faiblesses

  • Plugin Vite communautaire, non officiel ; passer à Rspack/Webpack serait un chantier majeur.
  • Runtime de fédération chargé sur chaque app (30 à 50 Ko bootstrap).
  • Modèle mental dense : exposes, remotes, sharedScope, eager, singleton, strictVersion…
  • Debug réputé opaque : erreurs de scope difficiles à isoler.
  • Couplage build-runtime fort, vendor lock-in sur le bundler.
  • Pour partager une lib (et pas orchestrer des micro-frontends), c'est utiliser un bazooka.
Approche 03 / 06

SystemJS

Obsolète

Le prédécesseur des import maps. Les modules sont encapsulés dans un wrapper System.register().

SystemJS a longtemps été la seule solution pour faire l'équivalent d'un import map dans des navigateurs sans support natif. Sa pertinence s'est effondrée avec l'arrivée des import maps natives en 2023. Il subsiste principalement comme dépendance de Single-SPA.

Forces

  • Compatibilité avec d'anciens navigateurs (IE11 si requis).
  • Mature, utilisé en production depuis presque dix ans.
  • Résolveur très flexible : hooks personnalisables.

Faiblesses

  • Remplacé par les import maps natives : aucun bénéfice technique sur navigateurs modernes.
  • Runtime de 5 à 10 Ko chargé sur chaque page.
  • Format de bundle non standard : transformation au build via rollup-plugin-systemjs.
  • Ne résoudrait pas vos blockers actuels (SVG, duplication React).
Approche 04 / 06

UMD & globals

Régression

Un <script> charge EFF, qui s'expose sur window.EFF. Le retour à l'approche jQuery.

Approche historique du partage de librairies : la lib se charge via une balise script, expose ses composants sur l'objet global, les apps les consomment via window.EFF.Button. Très simple en surface, mais c'est un retour en arrière par rapport au modèle ESM.

Forces

  • Simplicité maximale : pas d'import map, pas de plugin, pas de runtime.
  • Compatibilité universelle.
  • Cache HTTP trivial.

Faiblesses

  • Pollution du global, risque de collision, sécurité réduite.
  • Aucun tree-shaking : chargement intégral d'EFF même pour trois composants.
  • React doit aussi être global, double risque de duplication.
  • Support TypeScript pénible : déclarations à maintenir manuellement.
  • ESM est devenu le standard ; revenir à UMD est culturellement régressif.
Approche 05 / 06

Single-SPA

Hors périmètre

Framework de micro-frontends : un shell orchestre le mount/unmount d'applications complètes.

Single-SPA résout l'orchestration de plusieurs applications dans un même shell HTML : routing partagé, lifecycle de mount/unmount, partage de contexte. C'est complémentaire des import maps (souvent utilisé conjointement), pas une alternative.

Pour Edifice, ce n'est pas le besoin : l'agrégation est déjà assurée par Vert.x, on ne cherche pas à partager du routing ou du state inter-apps. On cherche à partager une lib.

Forces

  • Solution complète aux problématiques de micro-frontends.
  • Mature, utilisé en production chez American Express, etc.
  • Bonne intégration avec les import maps en sous-jacent.

Faiblesses

  • Réécrit l'architecture d'agrégation ; Vert.x devient un shell Single-SPA.
  • Lifecycle d'apps complexe à maîtriser.
  • Surdimensionné : réponse à un problème que vous n'avez pas.
  • Réserver pour une éventuelle réorganisation future inter-apps.
Approche 06 / 06

CDN avec versions pinées

Hors objectif

Chaque app code en dur l'URL de la version d'EFF qu'elle consomme.

Variante "naïve" de l'externalisation : au lieu d'un import map central, chaque app embarque dans son bundle des URLs pinées vers une version précise d'EFF (par exemple https://cdn.edifice.io/eff/2.5.19/index.js). Le bundler les laisse passer, le navigateur les charge.

Forces

  • Encore plus simple qu'un import map : pas de JSON central.
  • Chaque app maîtrise totalement sa version.

Faiblesses

  • N'atteint pas l'objectif principal : changer de version d'EFF impose de rebuilder chaque app, donc reproduit le problème actuel.
  • Versions multiples inévitables entre apps → duplication garantie de React.
  • Pas de mécanisme de mise à jour centralisé.

Recommandation pour Edifice

L'approche retenue doit s'aligner sur la stack en place, sur le besoin réel, et sur la durabilité de la décision.

Décision technique recommandée Import Maps natives

Cinq raisons spécifiques à votre contexte

Le choix n'est pas générique : il découle directement de la stack, des contraintes et de l'état d'avancement actuel du chantier ENABLING-722.

  1. Vous êtes 100 % Vite. Module Federation imposerait une migration vers Webpack ou un plugin Vite communautaire instable. C'est un chantier hors de proportion par rapport au bénéfice.
  2. L'infrastructure existe déjà. Nginx, Jenkins et le dossier recette/assets/js versionné sont parfaits pour servir des modules ESM. Aucun composant nouveau à déployer.
  3. Le besoin est le partage d'une lib, pas l'orchestration de micro-frontends. Single-SPA résoudrait un problème que vous n'avez pas. L'agrégation par Vert.x est suffisante.
  4. Standard W3C, donc pérenne. Pas de plugin tiers à maintenir, pas de risque qu'une bibliothèque devienne non maintenue dans cinq ans.
  5. Les blockers actuels sont réparables. La régression SVG est une regex à corriger ; la duplication React suspectée est un audit de trois packages. Aucune limite architecturale en jeu.

Le plan de remédiation technique pour engager cette approche — quatre étapes, ordonnées par criticité et par coût croissant — est développé dans la section suivante avec les extraits de code des correctifs.

Note Cette décision est révisable si un besoin de micro-frontends à part entière émerge plus tard (mount/unmount d'apps complètes, routing partagé, state inter-apps). Single-SPA reste compatible avec une base import-maps déjà en place, et constituerait la suite logique sans renoncer aux acquis.

Plan de remédiation technique

Quatre étapes concrètes, avec les fichiers concernés, les extraits de code des problèmes identifiés sur la branche feat-ENABLING-722, et les correctifs proposés.

Étape 01

Restaurer l'inlining des SVG dans le bundle d'EFF

Régression à corriger · 1 ligne modifiée · Effort : 10 min · Risque : nul

Une regex de plus que permissive transforme 483 SVG en requêtes HTTP qui échouent au parsing module.

Symptôme

Au chargement d'une page utilisant un composant d'empty state, le navigateur émet une cascade d'erreurs MIME pour chaque SVG :

⎯ Browser console
Failed to load module script: Expected a JavaScript module script
but the server responded with a MIME type of "image/svg+xml".
Strict MIME type checking is enforced for module scripts per HTML spec.

// URL en cause :
GET /assets/js/edifice-bootstrap/images/one/illu-empty-search-blog.svg
    net::ERR_FAILED  200
GET /assets/js/edifice-bootstrap/images/one/illu-empty-search-collaborativewall.svg
    net::ERR_FAILED  200
// ...répété pour chaque SVG importé par EFF

Source du problème

Le module emptyscreen-mapping d'EFF importe 23 SVG par leur chemin :

packages/react/src/utilities/emptyscreen-mapping/emptyscreen-mapping.ts TypeScript 
import illuOneBlog            from '@edifice.io/bootstrap/dist/images/one/illu-empty-search-blog.svg';
import illuOneCollaborativeWall from '@edifice.io/bootstrap/dist/images/one/illu-empty-search-collaborativewall.svg';
import illuOneCommunity        from '@edifice.io/bootstrap/dist/images/one/illu-empty-search-community.svg';
// ... 20 imports supplémentaires

Historiquement, ces imports étaient traités par @rollup/plugin-url({ limit: Infinity }) qui les inlinait comme data URIs dans le bundle d'EFF. Le commit b0e991ed8 a élargi la regex d'externalisation à tout le sous-arbre @edifice.io/bootstrap/*, y compris dist/images/* :

packages/react/vite.config.ts État actuel 
  build: {
    rollupOptions: {
      external: [
        ...Object.keys(dependencies ?? {}),
        ...Object.keys(peerDependencies ?? {}),
        'react/jsx-runtime',
        '@edifice.io/client',
        /^@edifice\.io\/tiptap-extensions\/.*/,
        /^@edifice\.io\/bootstrap\/.*/,  // ← rend les SVG external
        /^dayjs\/plugin\/.+\.js$/,
        /^dayjs\/locale\/.+\.js$/,
        /^antd\/locale\/.+/,
        /^swiper\/.*/,
      ],
    },
  },

Conséquence : les imports *.svg deviennent des spécifieurs bare, conservés tels quels dans le bundle final. À l'exécution, le navigateur consulte l'import map, résout vers une URL .svg, tente de la charger comme module ESM, et refuse à cause du MIME type non-JS.

Correctif

Restaurer le lookahead négatif qui exclut dist/images/ du périmètre d'externalisation :

packages/react/vite.config.ts Diff 
  external: [
    ...Object.keys(dependencies ?? {}),
    ...Object.keys(peerDependencies ?? {}),
    'react/jsx-runtime',
    '@edifice.io/client',
    /^@edifice\.io\/tiptap-extensions\/.*/,
    /^@edifice\.io\/bootstrap\/.*/,
    /^@edifice\.io\/bootstrap\/(?!dist\/images\/).*/,
    /^dayjs\/plugin\/.+\.js$/,
    // ...
  ],

Les chemins @edifice.io/bootstrap/dist/images/*.svg redeviennent internes au bundle d'EFF, donc traités par le plugin url() du tableau plugins qui les transforme en data URIs.

Validation

$ pnpm --filter @edifice.io/react build
✓ built in 18.2s

$ grep -c "data:image/svg" \
    packages/react/dist/utilities/emptyscreen-mapping/emptyscreen-mapping.js
22   # ← chaque SVG est désormais inliné

$ grep "bootstrap/dist/images" packages/react/dist/**/*.js
(no output)   # ← plus aucun import SVG résiduel dans le bundle
Étape 02

Auditer et corriger les sources de duplication de React

Investigation puis fix conditionnel · 3 packages à vérifier · Effort : 1 à 4 h · Risque : moyen

Tant que toutes les apps et leurs deps transitives résolvent react vers la même URL d'import map, il y a une seule instance. Trois suspects à éliminer.

Symptôme

Erreurs intermittentes côté apps consommatrices, hooks qui ne déclenchent pas de rerender, alertes dans la console :

⎯ Browser console
Warning: Invalid hook call. Hooks can only be called inside of the body
of a function component. This could happen for one of the following reasons:
  1. You might have mismatched versions of React and the renderer
  2. You might be breaking the Rules of Hooks
  3. You might have more than one copy of React in the same app

Architecture cible

Pour garantir l'instance unique, toutes les chaînes d'import doivent converger vers le même fichier physique. L'import map garantit déjà cela pour les apps ; reste à valider les libs intermédiaires.

recette/assets/importmap.json État actuel 
{
  "imports": {
    "@edifice.io/react":    "/assets/js/edifice-react/index.js",
    "react":               "/assets/js/edifice-react/react-esm/react.js",
    "react-dom":           "/assets/js/edifice-react/react-esm/react-dom.js",
    "react/jsx-runtime":   "/assets/js/edifice-react/react-esm/jsx-runtime.js",
    "ode-explorer":        "/assets/js/ode-explorer/dist/lib/index.js",
    "@edifice.io/client":   "/assets/js/edifice-client/index.js"
  }
}

Suspect n°1 — @edifice.io/client avec external désactivé

Sa configuration Vite a la ligne d'externalisation commentée, ce qui inline toutes les dépendances dans le bundle final. Si l'une d'elles transitivement importe React, il y a un second exemplaire.

packages/client/vite.config.ts État actuel · lignes 10-20 
  build: {
    lib: {
      entry: resolve(__dirname, './src/ts/index.ts'),
      name: '@edifice.io/client',
      fileName: 'index',
      formats: ['cjs', 'es'],
    },
    rollupOptions: {
      //external: [...Object.keys(dependencies)],  ← désactivé
    },
  },

Audit :

$ grep -E "(React\.createElement|jsx-runtime|useState)" \
       packages/client/dist/index.es.js | head -3
# Si la sortie est non-vide → client embarque du code React → fix requis

$ pnpm --filter @edifice.io/client list --depth 5 | grep "react"
# Identifie les chemins de dépendance vers react / react-dom

Correctif : forcer l'externalisation de React quoi qu'il advienne des autres deps.

packages/client/vite.config.ts Diff 
    rollupOptions: {
      //external: [...Object.keys(dependencies)],
      external: [
        ...Object.keys(dependencies ?? {}),
        /^react($|\/)/,
        /^react-dom($|\/)/,
      ],
    },

Suspect n°2 — ode-explorer

Lib externe sur laquelle on a moins de contrôle. Vérifier qu'elle ne bundle pas React au lieu de le déclarer en peerDependencies :

$ cd /Volumes/Work/recette/assets/js/ode-explorer
$ grep -c "jsxRuntime\|React.createElement" dist/lib/index.js
3247   # Si très élevé → React bundlé en dur

$ cat package.json | jq '.peerDependencies, .dependencies'
# React doit être en peerDependencies, pas dependencies

Si l'audit confirme le bundling : faire corriger en amont (peer dependency + rebuild), ou en patch local via pnpm patch. Pas de workaround côté apps possible : une fois React inliné dans ode-explorer, plus moyen de le partager.

Suspect n°3 — @edifice.io/react lui-même

La configuration en place compile React via deux configs Vite secondaires (reactEsmConfig, reactDomEsmConfig) qui inlinent React entièrement dans un fichier servi par l'import map. C'est le mécanisme correct, à condition que le shim ne soit pas accidentellement bypassé :

packages/react/src/shims/react.ts À vérifier 
export * from 'react';
export { default } from 'react';

Et la config qui le compile (depuis vite.config.ts, à la fin du fichier) :

packages/react/vite.config.ts extrait — reactEsmConfig 
export const reactEsmConfig = defineConfig({
  build: {
    lib: {
      entry: { react: resolve(__dirname, 'src/shims/react.ts') },
      formats: ['es'],
      fileName: () => 'react.js',
    },
    outDir: 'dist/react-esm',
    rollupOptions: {
      external: [],  // ← critique : doit rester vide pour inliner React
    },
  },
  plugins: [react({ babel: { plugins: ['@babel/plugin-transform-react-pure-annotations'] } })],
});

Validation

Après correctifs, tester depuis la DevTools Console d'une page blog en recette :

> const r1 = await import('react');
> const r2 = await import('@edifice.io/react');
> console.log(r1.version, Object.is(r1.useState, window.React?.useState));

Étape 03

Stratégie de versioning multi-app via scopes

Évolution structurelle · Import map + pipeline Jenkins · Effort : 1 jour · Risque : faible

Préparer dès maintenant la cohabitation contrôlée de plusieurs versions d'EFF, sans dupliquer React.

Problème adressé

Lors d'une migration d'EFF (rupture d'API, montée majeure), blog peut avoir besoin de la version 2.6 pendant que les autres apps restent en 2.5. L'import map actuel n'a qu'un seul niveau de résolution global.

État actuel

Chemins courts non versionnés :

recette/assets/importmap.json JSON · v1 
{
  "imports": {
    "@edifice.io/react":    "/assets/js/edifice-react/index.js",
    "@edifice.io/react/":   "/assets/js/edifice-react/",
    "@edifice.io/client":   "/assets/js/edifice-client/index.js",
    "react":               "/assets/js/edifice-react/react-esm/react.js",
    "react-dom":           "/assets/js/edifice-react/react-esm/react-dom.js",
    "ode-explorer":        "/assets/js/ode-explorer/dist/lib/index.js"
  }
}

État cible

Chemins versionnés explicitement, scope de surcharge pour blog en migration. La clé : on scope EFF mais on garde React partagé en singleton.

recette/assets/importmap.json JSON · v2 (avec scopes) 
{
  "imports": {
    // EFF : version par défaut pour toutes les apps
    "@edifice.io/react":    "/assets/js/eff/2.5.19/react/index.js",
    "@edifice.io/react/":   "/assets/js/eff/2.5.19/react/",
    "@edifice.io/client":   "/assets/js/eff/2.5.19/client/index.js",

    // React : partagé, jamais scopé
    "react":               "/assets/js/shared/react/18.3.1/react.js",
    "react-dom":           "/assets/js/shared/react/18.3.1/react-dom.js",
    "react/jsx-runtime":   "/assets/js/shared/react/18.3.1/jsx-runtime.js",

    "ode-explorer":        "/assets/js/ode-explorer/develop-pedago/index.js"
  },
  "scopes": {
    // Blog épingle EFF 2.6.0 pendant sa migration ; React reste hérité du global.
    "/blog/": {
      "@edifice.io/react":  "/assets/js/eff/2.6.0/react/index.js",
      "@edifice.io/react/": "/assets/js/eff/2.6.0/react/",
      "@edifice.io/client": "/assets/js/eff/2.6.0/client/index.js"
    }
  }
}

Le scope "/blog/" ne surcharge que les entrées EFF. Quand un module chargé sous /blog/... importe react, le navigateur remonte au niveau global et trouve l'unique react.js. Le singleton est préservé.

Contrainte Les versions cohabitantes d'EFF doivent rester compatibles avec la même version de React. Une rupture React (18 → 19) ne peut pas se faire en cohabitation : c'est une bascule synchronisée de tout l'écosystème, traitée à part.

Impact sur le pipeline Jenkins

Le script de build de recette doit produire des dossiers versionnés au lieu d'écraser un emplacement unique. Modification du build.sh :

Sanctuariser la configuration Vitest des apps consommatrices

Stabilisation des tests · 2 fichiers · Effort : 30 min · Risque : nul

Le plugin external-libs est désactivé en test (apply: 'build'), mais il faut un filet de sécurité pour les assets non-JS.

Le mécanisme actuel

Le plugin d'externalisation de blog est correctement borné :

Le risque résiduel

Si un test charge directement la source d'EFF (ou si inline force Vitest à retraverser des modules incluant des imports *.svg), Vitest n'a pas de handler pour les SVG et émet une erreur. La sécurité consiste à fournir un stub global pour tous les assets non-JS.

Correctif

blog/frontend/vite.config.ts Diff · bloc test 
  test: {
    watch: false,
    globals: true,
    environment: 'jsdom',
    setupFiles: ['./src/mocks/setup.ts'],
    server: {
      deps: {
        inline: ['@edifice.io/react'],
        inline: [/@edifice\.io\//, 'ode-explorer'],
      },
    },
    alias: [
      {
        find: /\.svg(\?.*)?$/,
        replacement: resolve(__dirname, './src/mocks/svg-stub.ts'),
      },
      {
        find: /\.(png|jpe?g|gif|webp)$/,
        replacement: resolve(__dirname, './src/mocks/asset-stub.ts'),
      },
    ],
  },

Récapitulatif et ordre d'exécution

Engager les étapes dans l'ordre 01 → 04. Chacune débloque la suivante en réduisant le bruit de signal :

  1. Étape 01 répare la régression la plus visible et restaure un état build correct.
  2. Étape 02 ne peut être conduite proprement qu'une fois 01 stabilisée (sinon les erreurs SVG masquent les vraies erreurs React).
  3. Étape 03 est une évolution architecturale : à engager une fois 01-02 verts, pour préparer la première vraie migration multi-version.
  4. Étape 04 peut se faire en parallèle de 01 et 02 ; c'est un travail de stabilisation indépendant côté apps.
