1 · Patterns React

12 findings · 0 React.memo, 108 useMemo/useCallback, 134 forwardRef, 378 useState, 26 createContext

1.1 Chaînes de useState dans les hooks de modales/formulaires

Haut
  • packages/react/src/modules/modals/PublishModal/hooks/usePublishModal.tsx:36-60
  • packages/react/src/modules/modals/ShareModal/ShareResources.tsx:220-221

5–7 useState interdépendants par hook : transitions fragiles, ordre implicite, callbacks qui se chaînent sans orchestration.

Regrouper en useReducer avec actions nommées ; rend les transitions traçables et facilite l'ajout de tests.

1.2 exhaustive-deps désactivé sans corriger la cause

Haut
  • packages/react/src/components/TreeView/hooks/useTreeView.ts — 5 disable inline
  • packages/react/src/hooks/useThumbnail/useThumbnail.ts — disable de fichier
  • packages/react/src/hooks/useDropdown/useDropdown.ts — disable de fichier
  • + 15 warnings actifs (cf. pnpm -F @edifice.io/react lint)

Les eslint-disable masquent des dépendances circulaires et un manque de mémoïsation de callbacks parents : bugs silencieux quand les props/values changent.

Reprendre fichier par fichier : extraire les références stables, useCallback côté parent, ou useEvent-like pour les handlers ; supprimer les disables.

1.3 Aucun React.memo dans 1 106 fichiers

Haut
  • packages/react/src/modules/modals/ShareModal/ShareResources.tsx:365-403 — ShareBookmarkLine
  • packages/react/src/components/TreeView/TreeNode.tsx — récursion non mémoïsée
  • packages/react/src/components/UserRightsList/UserRightsList.tsx
  • packages/react/src/components/Layout/components/Header.tsx — 14+ hooks de données

Les composants de liste et de cellule re-rendent à chaque changement d'un sibling ; coûteux sur les modales de partage (N droits × M lignes).

Ajouter memo(forwardRef(...)) ciblé sur les items de liste et les sous-composants stables (Header, TreeNode) avec comparateur shallow.

1.4 Retour de hooks incohérent (tuple vs objet, any)

Moyen
  • packages/react/src/hooks/useToggle/useToggle.ts:4 — retour [boolean, any]
  • packages/react/src/hooks/useCheckable/useCheckable.ts:47-53 — objet à 5 propriétés
  • packages/react/src/hooks/useImage/useImage.ts:19 — objet {imgSrc, onError}

Aucune convention publique : les consommateurs alternent destructuration positionnelle et nommée, le any annule TypeScript.

Convention : objet sauf pour les hooks "primitive + setter" type useToggle ; typer le setter avec React.Dispatch<SetStateAction<T>>.

1.5 Header charge 14+ hooks à chaque render

Moyen
  • packages/react/src/components/Layout/components/Header.tsx:48-90 — useConversation, useUser, useHasWorkflow×2, useHelp, useHeader

Le composant le plus lourd du framework (477 lignes) déclenche en cascade des queries et des dérivations ; aucune mémoïsation.

Mémoïser Header, extraire des sous-composants HeaderUserMenu, HeaderApps, ne fetcher que sur interaction (déjà partiel sur la modale d'aide).

1.6 État dérivé non mémoïsé

Moyen
  • packages/react/src/modules/modals/PublishModal/hooks/usePublishModal.tsx:171-186 — listes retournées telles quelles
  • packages/react/src/components/TreeView/hooks/useTreeView.ts:54 — selectedNodeId recalculé à chaque render

Identité de référence change à chaque render → propagé aux useEffect et useCallback qui s'exécutent en cascade.

Wrap dans useMemo ; en parallèle, audit des dépendances d'effets en aval.

1.7 Prop drilling profond dans TreeView

Moyen
  • packages/react/src/components/TreeView/TreeNode.tsx:61-71 — 11 props (callbacks only)
  • packages/react/src/components/TreeView/TreeView.tsx:72-100 — destructure 7 props, passe à un hook qui passe à TreeNode

Aucune composition (compound, slots, render-prop, contexte interne) : impossible d'ajouter virtualisation, custom render, drag handle sans casser l'API publique.

Introduire TreeContext pour les callbacks et un slot de rendu personnalisable (renderNode).

1.8 Cleanup d'event listeners hétérogène

Moyen
  • packages/react/src/hooks/useZendeskGuide/useZendeskGuide.tsx:193 — window.addEventListener sans cleanup explicite dans l'effet imbriqué
  • packages/react/src/hooks/useHover/useHover.ts:16-17 — pattern correct mais dupliqué ailleurs

Quelques effets oublient le cleanup quand les deps changent ; fuite mémoire sur apps long-lived (PWA).

Mutualiser dans un useEventListener(target, type, handler, deps) qui impose le cleanup ; remplacer progressivement les usages directs.

1.9 Trois hooks d'état qui se chevauchent dans ShareResources

Moyen
  • packages/react/src/modules/modals/ShareModal/ShareResources.tsx:223-269 — useShare, useSearch, useShareBookmark indépendants

Aucun contexte unificateur : risque de désynchronisation entre shareRights et bookmark ; logique d'ordonnancement dans le composant.

Soit ShareContext unique + useReducer, soit collapser les 3 hooks en un useShareModalState qui orchestre.

1.10 Providers : couplage à antd et imbrication

Moyen
  • packages/react/src/providers/AntThemeProvider
  • packages/react/src/providers/EdificeThemeProvider
  • packages/react/src/providers/EdificeClientProvider

Le thème est partagé entre Antd et "Edifice" : 2 providers à maintenir en parallèle. Tout consommateur paie Antd côté runtime même s'il n'utilise pas les composants Antd-based.

Audit des usages d'Antd → si < 10 composants, viser un remplacement progressif ; sinon, fusionner les providers de thème en un seul.

1.11 Hook "god" : usePublishModal

Bas
  • packages/react/src/modules/modals/PublishModal/hooks/usePublishModal.tsx:32-187 — form + image + activities + API

160 lignes mélangeant react-hook-form, manipulation d'image, sélection métier et appels API : intestable en isolation.

Découper : usePublishForm, usePublishImage, usePublish ; composer dans le composant.

1.12 Frontière modules vs components floue

Bas
  • packages/react/src/modules/editor/components/* — composants potentiellement réutilisables
  • packages/react/src/modules/comments/components/* — idem

Aucune règle claire : modules = feature à état métier ? components = UI pure ? Risque de duplication si on ajoute des features.

Documenter la règle dans CONTRIBUTING.md et un test d'architecture (eslint-plugin-boundaries ou dependency-cruiser).

Points forts à conserver

  • Organisation hook-par-dossier (46 dossiers) : lisible, testable, refactor facile.
  • Patterns useKeyPress, useTrapFocus, useClickOutside : cleanup propre, à utiliser comme modèle.
  • forwardRef utilisé largement (134 occurrences) avec useImperativeHandle pour les APIs critiques (ShareResourcesRef, ComboboxRef).
  • Antd cantonné à quelques composants (MediaViewer/Carousel, Table, DatePicker) — pas un wrap global.
  • Babel @babel/plugin-transform-react-pure-annotations activé : aide rollup pour le tree-shaking.

2 · TypeScript

strict true sur @edifice.io/react · tsc --noEmit passe sans erreur · 330 any · 5 @ts-ignore/@ts-expect-error

2.1 Concentration de any dans @edifice.io/client

Haut
  • packages/client/src/ts/transport/Http.ts — 28 occurrences
  • packages/client/src/ts/transport/Service.ts — 15
  • packages/client/src/ts/session/interfaces.ts — 15
  • packages/client/src/ts/configure/interfaces.ts — 15

Les interfaces publiques (IConfigurationFramework, transport HTTP) typent les payloads en any : tout consommateur hérite du any.

Génériquer : HttpService.get<T>(...): Promise<T>, getPublicConf<T>(). Faisable sans breaking change avec un défaut = unknown.

2.2 tsconfig de packages/client non aligné

Moyen
  • packages/client/tsconfig.json (et autres packages hors react)

Le package react a strict + noUnusedLocals + noUnusedParameters ; les autres packages héritent d'une config plus laxiste — d'où la concentration de any.

Créer packages/config/tsconfig.strict.json et l'imposer à tous ; mettre à niveau client/utilities/extensions en plusieurs PRs.

2.3 Génériques manqués sur composants polymorphes

Moyen
  • packages/react/src/components/Dropdown/Dropdown.tsx:38 — children: (...props: any) => ReactNode
  • packages/react/src/components/Combobox/Combobox.tsx:57 — icon?: any

Les slots et render-props perdent leur type ; pas d'autocomplétion, casts manuels.

Typer les slots avec le ContextValue exposé, et icon: ReactNode ou union explicite.

2.4 @ts-ignore dans des manipulations DOM

Moyen
  • packages/react/src/components/Menu/hooks/useMenu.ts — 3 occurrences
  • packages/react/src/components/Layout/hooks/useHelp.tsx — 1

Camouflent des appels setAttribute sur des nœuds typés trop largement ; tout refactor casse silencieusement.

Typer currentNode/previousNode en HTMLElement | null et garde-fou explicite.

2.5 Types publics non centralisés pour modules/icons

Bas
  • packages/react/src/modules/icons/components/index.ts — 356 re-exports

Pas de type IconComponent exposé : un consommateur ne sait pas si IconAdd est React.FC<SVGProps> ou un composant avec ref.

Exposer export type IconComponent = React.FC<SVGProps<SVGSVGElement>> et générer les .tsx SVGR avec cette signature.

3.1 Module icons : 356 fichiers, 3 Mo

Haut
  • packages/react/src/modules/icons/components/ — 356 .tsx (apps, audience, nav, ...)
  • Déjà splitté en 4 entry points (icons, icons-nav, icons-apps, icons-audience)

Sans preserveModules, importer une seule icône paierait tout. Avec, on a 356 chunks à parser. C'est ce qui force l'architecture preserveModules sur tout le package.

Option A : générer un sprite SVG unique + composant <Icon name="..." />. Option B : maintenir les exports mais sortir icons dans un sous-package @edifice.io/icons avec son propre bundler — le reste du package peut alors abandonner preserveModules.

3.2 Module editor non lazy

Haut
  • packages/react/src/modules/editor/* — 40+ extensions TipTap
  • packages/react/package.json — toutes en dependencies

TipTap charge ProseMirror + 40 packages d'extensions ; toute app qui importe quoi que ce soit d'@edifice.io/react/editor paie 512 Ko (+ peerdeps).

Côté FF : déjà bien isolé en ./editor. Côté apps : documenter et imposer const Editor = lazy(() => import('@edifice.io/react/editor')). Vérifier que Vite/Rollup downstream split bien.

3.3 preserveModules: true en prod = 781 chunks ESM

Moyen
  • packages/react/vite.config.ts:90-92

Tree-shaking maximal mais HTTP/2 ou pas, 781 requêtes/résolutions de modules en plus côté apps. Discutable si les apps re-bundlent (Vite).

Mesurer : build:analyze existe déjà (stats.html). Si re-bundlé en aval, conserver. Sinon, basculer sur sortie standard pour les modules autres que icons.

3.4 sideEffects: false manquant sur 3 packages

Moyen
  • packages/utilities/package.json
  • packages/client/package.json
  • packages/extensions/package.json

Sans ce flag, les bundlers consommateurs ne shakent pas le code non utilisé — gaspillage de bundle côté apps Edifice.

Ajouter "sideEffects": false (ou liste précise pour bootstrap qui contient des SCSS). 30 min de vérification, gain immédiat.

3.5 esbuild.minifyIdentifiers: false en prod

Bas
  • packages/react/vite.config.ts:14

Choix conscient pour la lisibilité des stacktraces consommateurs, mais coût en taille (~10-15 % sur le JS). À évaluer si les apps re-minifient elles-mêmes.

Vérifier le pipeline des apps consommatrices : si elles re-minifient en aval, ce flag est neutre côté prod. Sinon, l'inverser pour les builds non-analyze.

B.1 Breakpoints désynchronisés entre SCSS et hook TypeScript

Haut
  • packages/bootstrap/src/themes/configs/_breakpoints.scss — mobile 360, mobile-large 390, tablet 720, desktop-small 960, desktop 1264, desktop-large 1320
  • packages/react/src/hooks/useBreakpoint/useBreakpoint.ts:14-21 — xs 0, sm 375, md 768, lg 1024, xl 1280, xxl 1400 (breakpoints Bootstrap par défaut)

Deux systèmes différents : noms ET valeurs. Une app qui combine useBreakpoint().md (≥768px) avec un @include respond-from('tablet') SCSS (≥720px) aura un trou de 48px où JS et CSS divergent — bug perçu comme "magique" par les équipes.

Unifier en source unique. Option recommandée : exposer les breakpoints Sass via CSS variables (--breakpoint-md: 768px) et lire ces variables depuis le hook TS — ou inverser, depuis un JSON partagé. Documenter dans THEMING.md.
